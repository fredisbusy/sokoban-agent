# 프로젝트 목표

> 처음 보는 Sokoban 보드에서 정답 경로를 미리 받지 않고, 공간 구조를
> 분석해 가설과 하위 목표를 세우고, 실행 결과를 관찰해 계획을 수정하는
> AI 에이전트를 단계적으로 연구한다. LangGraph는
> `관찰 → 분석 → 계획 → 검증 → 실행 → 성찰`을 오케스트레이션하고,
> 규칙과 탐색 알고리즘은 에이전트의 결정을 대신하지 않는 검증·도구·평가
> 기준으로 사용한다.

여기서 “사람처럼 문제를 푼다”는 숨은 사고 과정이나 인간 인지의 재현을
뜻하지 않는다. 보드 분석, 상자-목표 배정, 전략 가설, 보호할 칸, 현재 하위
목표, 예상 효과, 실패 조건과 수정 이력을 관찰 가능한 구조로 남기고 실제
행동 선택에 사용하는 문제 해결 방식을 뜻한다.

## 연구 질문

- 구조화된 보드만 보고 상자-목표 배정, push 순서와 공간 확보 같은
  장기 전략을 명시적으로 세울 수 있는가?
- 구조화된 하위 목표가 원시 방향 행동을 바로 생성하는 LLM보다 처음 보는
  보드 구조에 더 잘 일반화하는가?
- 에이전트가 선언한 가설과 위험이 실제 행동 선택과 후보 제거에 인과적으로
  사용되는가?
- 실패했을 때 전체 계획을 버리지 않고 관련 하위 목표만 진단하고 수정할 수
  있는가?
- 국소 탐색 도구가 에이전트의 전략을 대신하지 않으면서 성공률과 효율을
  높이는가?
- 전역 bounded A* oracle과 비교했을 때 성공률, 행동·push 비용, 탐색량과
  모델 호출 비용의 교환 관계는 무엇인가?
- 상태 기반 에이전트가 안정된 뒤 화면 기반 인식과 장기 기억으로 확장할 수
  있는가?

## 범위

- 자체 Gymnasium 환경과 Boxoban 레벨을 동일한 규칙으로 실행한다.
- 보드 분석, 전략 가설, 상자-목표 배정, 보호 제약과 한 번에 하나의 하위
  목표를 구조화된 계획 산출물로 정의한다.
- 에이전트는 한정된 행동 또는 push만 실행한 뒤 반드시 새 상태를 관찰한다.
- 규칙 엔진은 합법성, 성공과 명백한 데드락을 결정론적으로 판정한다.
- 플레이어 도달성이나 선택한 하위 목표의 세부 동선에는 국소 탐색을 사용할
  수 있지만, 전역 A* 정답 경로는 주 에이전트 실행 중 제공하지 않는다.
- Random, 원시 행동 LLM, 구조화된 하위 목표 LLM, 국소 탐색 결합 정책과
  전역 A* oracle을 동일한 고정 코호트에서 비교한다.
- 성공률뿐 아니라 하위 목표 달성, 배정 변경, 가설 수정, 설명-행동 일치,
  데드락, 비용과 일반화 성능을 기록한다.

대규모 강화학습, 인간과 동일한 숨은 사고 과정의 재현, 처음부터 전체
Boxoban을 완전하게 푸는 산업용 solver, 화면 인식과 장기 기억은 현재
핵심 범위가 아니다.

## 설계 원칙

- 게임 규칙과 판정의 기준은 환경 코드 하나로 유지한다.
- LangGraph `StateGraph`가 계획 산출물, 에피소드 상태, 재시도, 조건부 전이,
  체크포인트와 subgraph 수명 주기를 소유한다. 이를 감싸는 자체 workflow
  framework나 상태 머신을 만들지 않는다.
- 프롬프트 조립과 모델 호출은 LangGraph node로 드러내고, 프롬프트 본문과
  버전은 LangSmith Prompt Management를 우선 사용한다. 실행 state에는
  prompt 이름·commit과 입력 변수만 기록하며 자체 registry·cache·배포
  체계를 만들지 않는다.
- 전략 Planner는 원시 행동열보다 보드 분석과 실행 가능한 하위 목표를
  제안한다.
- 계획 산출물의 Interface에는 가설, 근거, 보호 제약, 예상 효과와 실패
  조건을 포함하고, 검증과 계측은 이 Interface를 공통 시험면으로 사용한다.
- 규칙 검증 Module은 전략을 만들지 않고 합법성과 명시된 제약만 판정한다.
- 국소 탐색 Module은 선택된 하위 목표의 세부 동선을 숨기는 깊은 Module로
  두며, 전역 문제를 대신 풀지 않는다.
- 전역 A*는 실행 정책과 분리된 평가 seam 뒤의 oracle Adapter로 유지한다.
- 모델의 자연어 설명만으로 문제 해결을 주장하지 않는다. 구조화된 산출물을
  제거하거나 변경했을 때 행동도 달라지는지 ablation으로 확인한다.
- 노트북은 탐색과 시각화에만 쓰고 재사용 코드는 `src/`에 둔다.
- 직접 작성하는 Python 파일은 400줄 이하로 유지하고, 넘기기 전에 서로 다른
  책임을 별도 모듈로 분리한다.
- seed, 설정, 결과를 남겨 같은 실험을 다시 실행할 수 있게 한다.
- 작은 고정 퍼즐에서 계획 산출물의 의미를 검증한 뒤 구조가 다른 held-out
  레벨로 확장한다.

## 성공 기준

1. 에이전트가 매 계획 시점에 검증 가능한 보드 분석, 가설과 현재 하위
   목표를 구조화해 남긴다.
2. 실행된 행동은 현재 하위 목표와 연결되고 예상 효과의 충족 여부가 다음
   관찰에서 판정된다.
3. 실패 시 관련 가설 또는 하위 목표를 수정하고 같은 실패 계획의 반복을
   탐지한다.
4. 전역 A*의 정답 suffix나 전체 대체 없이 held-out 소형 레벨에서 반복
   가능한 성공 결과를 기록한다.
5. 원시 행동 LLM과 비교해 구조화된 계획의 성공률, 데드락 또는 복구 지표
   중 하나 이상에서 기여를 보이며, 국소 탐색 사용량과 모델 비용을 함께
   보고한다.
6. 변경을 완료할 때 `pytest`, Ruff, mypy 검사를 통과한다.

## 현재 상태

자체 Gymnasium 환경과 레벨 로더 위에 LangGraph 실행 코어가 구현됐다.
그래프의 `plan → validate → execute` 조건부 전이가 성공, 데드락, 행동 제한,
계획 오류와 막힌 행동 재시도를 관리한다. 각 에피소드는 `thread_id`로
체크포인트된다.

Random, BFS, push 기반 A*, ChatLiteLLM Planner가 같은 그래프를 사용한다.
알고리즘은 전체 경로를, LLM은 최대 8개 행동을 제안한다. 실행 결과에는
계획 호출·오류·재시도, 탐색 확장 수와 시간, LangChain 표준 입력·출력
토큰 지표가 함께 기록된다.

LangGraph Studio용 그래프는 `llm_plan → astar_guard → validate_plan →
execute_action`을 각각 관찰 가능한 노드로 제공한다. 노드와 상태 필드 이름은
영어이고, 모델 프롬프트와 목표·판단·위험·검사 요약 내용은 한국어이며 로컬
실행이 기본이다.

LLM 기여도를 분리하기 위한 다섯 정책 ablation과 연구 계측도 구현됐다.
고정된 Boxoban test 코호트에서 A* 단독, LLM 단독, suffix-only,
full-guard, always-replace를 비교한다. 에피소드 결과에는 push, 상태 재방문,
반복 제안, LLM prefix 채택률과 bounded A* reference 대비 행동·push·탐색량
차이가 포함된다. 외부 데이터의 커밋과 체크섬, 레벨 ID 목록은 manifest로
고정한다.

기존 원시 행동 실행과 `SearchGuardPlanner`는 새 방향의 비교 기준으로
보존한다. `SearchGuardPlanner`는 LLM 제안 뒤의 정답 suffix를 붙이거나 전체
계획을 교체할 수 있으므로, 그 성공 결과를 에이전트의 독립 문제 해결로
합산하지 않는다.

주 연구 정책은 하나의 compiled `StateGraph`에서 보드 분석, LangSmith
prompt commit 해석, 구조화 전략 제안·검증, 단일 push 접지, 실행, 관찰과
성찰을 반복한다. 실행 중 전역 A*를 참조하지 않으며, CLI·평가·Studio와
Agent Server가 같은 graph factory를 사용한다. prompt/model fixture로
schema 오류, 의미 오류, 반복 종료와 rationale ablation을 오프라인에서
검증한다.

구조 태그와 checksum을 가진 held-out manifest, 여섯 정책 비교 runner와
재현 가능한 일반화 노트북도 구현됐다. 규칙·도달성·국소 탐색 호출과
LLM 토큰·시간을 분리하며, action trajectory와 rationale 제거 전후의 행동
변화를 기록한다.

단순 행동 계획과 구조화 전략 결정은 각각 최대 512토큰으로 제한한다.
모델은 매 push마다 전체 전략을 다시 쓰지 않고 `push_id`, `target_id`,
보호할 칸과 짧은 근거·위험만 반환한다. 목적지, 예상 효과, 실패 조건과
검증 가능한 `StrategyHypothesis`는 현재 `BoardAnalysis`에서 결정론적으로
합성한다. 두 예산은 연구 manifest에 함께 기록해 정책 비용 비교에서 설정
차이를 숨기지 않는다.

private LangSmith `sokoban-strategy` prompt의 commit
`731d3f516b225cc0e1d11b87cfc5abe45c1a92ed63b41ef3e23796e805006b77`를
compact push decision 계약으로 고정했다. 실제 모델로
`boxoban-unfiltered-76`을 47 actions, 10 pushes, schema·의미 거절 0회로
해결했다. 직전 push의 역방향 후보는 모델 입력에서 제외하고, 반복은 플레이어
좌표가 아니라 모델이 보는 보드 추상화와 하위 목표로 판정한다. 남은 연구
단계는 같은 commit을 고정한 held-out 여섯 정책 결과를 반복 실행해
일반화·비용 차이를 축적하는 것이다.

실제 일반화 코호트는 Boxoban 공식 `unfiltered`, `medium`, `hard`에서
결정적으로 5개씩 선정한 15개 보드다. 원본 commit과 파일 checksum,
인라인 payload checksum을 고정하고 bounded A*로 20만 상태 안에서 모두
해가 확인되는지 검증했다. Boxoban manifest와 직접 만든 JSON 맵은 하나의
생성 catalog로 합쳐진다. 웹은 `level_id`와 checksum을 전달하고 Agent
Server가 같은 catalog에서 보드를 해석하므로 연구와 관찰 화면의 맵 계약이
갈라지지 않는다.
